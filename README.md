# LinuxSwipe

LinuxSwipe - mobile-first тренажер Linux-команд на React Native и Expo. Приложение превращает практику терминала в короткие учебные сессии: пользователь видит сценарий, вспоминает команду, проверяет ответ и свайпом решает, знает он ее или хочет повторить позже.

Production: [linuxswipe.expo.app](https://linuxswipe.expo.app)

## Что это за продукт

Linux-команды легко узнавать в документации, но заметно сложнее быстро вспоминать в реальной задаче, на дежурстве или на собеседовании. LinuxSwipe построен вокруг active recall: сначала приложение просит достать команду из памяти, а уже потом показывает ответ, пример, пояснение и допустимые варианты.

Цель продукта не заменить `man` или документацию. Цель - помочь начинающим DevOps-инженерам, Linux-администраторам и support-специалистам набрать уверенность через короткие, повторяемые учебные подходы.

## Как проходит сессия

1. Пользователь получает карточку из текущей колоды.
2. Читает практический сценарий и пытается вспомнить команду.
3. Тапает по карточке, чтобы открыть ответ.
4. При необходимости открывает `Подробнее` с примером, подсказкой и объяснением.
5. Свайпает вверх, если команда уже знакома уверенно.
6. Свайпает вниз, если карточку нужно вернуть в повторение.
7. Может открыть ручной ввод и проверить ответ не по точному совпадению строки, а через matcher.

## Что уже реализовано

- 200 карточек по Linux-командам с категориями, сложностью и подробными объяснениями.
- Bottom-tab навигация между экранами `Карточки`, `Команды` и `Статистика`.
- Отдельный таб `Команды` со справочником, поиском, сортировкой и full-screen деталями по каждой Linux-команде из текущей колоды.
- Tap-to-flip сценарий с крупной карточкой и отдельным full-screen окном `Подробнее`.
- Ручной ввод ответа с более гибкой проверкой: учитываются допустимые варианты, форматирование и похожие команды.
- Свайпы `Знаю` и `На повтор` с учебной очередью повторения.
- Подмешивание review-карточек обратно в основной поток, чтобы повтор не жил отдельно от новой колоды.
- Фильтры по теме, сложности, подборке и режиму колоды.
- Готовые тематические подборки: `DevOps`, `Файлы и права`, `Сеть`, `Shell-практика`.
- Первый onboarding-экран с объяснением жестов и ручного ответа.
- Экран прогресса с опытом, уровнем, серией, дневной целью, точностью ручных ответов и прогрессом по темам.
- Локальное сохранение прогресса через `AsyncStorage`.
- Загрузка карточек из Supabase при наличии env-переменных.
- Кэш карточек на устройстве и fallback на локальный JSON, если Supabase недоступен или вернул невалидные данные.
- Web production deploy через EAS Hosting.

```text
    shared/
      api/
      components/

```

Проект организован по feature-based подходу. Все, что относится к учебному сценарию, лежит в `src/features/learning`, а общие API-, UI- и theme-части вынесены в `src/shared`.

Ключевые зоны внутри `src/features/learning`:

- `components/` - карточки, sheets, модалки, onboarding и toast feedback.
- `screens/` - основной учебный экран и экран аналитики прогресса.
- `hooks/` - логика свайпов, ручного ответа, toast-состояний и сборки текущей колоды.
- `context/` - провайдер учебного состояния, гидратация и сохранение прогресса.
- `lib/` - чистая бизнес-логика: deck builder, matcher команд, XP, даты, статистика.
- `data/` - локальные карточки, runtime validation и репозиторий для локального/Supabase источника.
- `storage/` - изоляция работы с `AsyncStorage` для прогресса, onboarding и card cache.

## Источник карточек и Supabase

Сейчас приложение поддерживает двухуровневую модель контента:

- базовый источник правды в репозитории - `src/features/learning/data/cards.json`;
- удаленный источник - таблица `cards` в Supabase.

Во время запуска приложение:

1. Загружает локальное сохраненное состояние пользователя.
2. Поднимает стартовый набор карточек из локального кэша или из `cards.json`.
3. Если Supabase настроен, фоном проверяет более свежую версию карточек.
4. При успешной загрузке валидирует remote-данные, кэширует их и обновляет текущую сессию.
5. Если Supabase недоступен, пустой или вернул плохие данные, продолжает работать на локальном JSON/кэше без поломки UX.

Это дает офлайн-устойчивый сценарий: UI, matcher, статистика и deck builder не завязаны на конкретный источник данных.

### Переменные окружения

```bash
EXPO_PUBLIC_SUPABASE_URL=...
EXPO_PUBLIC_SUPABASE_KEY=...
```

Шаблон лежит в `.env.example`. Файл `.env.local` добавлен в `.gitignore`.

### Подготовка таблицы

Схема и seed лежат в директории `supabase/`:

```sql
supabase/cards.schema.sql
supabase/cards.seed.sql
```

`cards.schema.sql` создает таблицу, индексы, RLS и public read-only policy для активных карточек.

Чтобы пересобрать SQL seed из текущего JSON:

```bash
npm run content:seed:supabase
```

Команда обновит `supabase/cards.seed.sql` и сохранит upsert по `id`, чтобы повторный импорт не создавал дубли.

## Web deploy

LinuxSwipe выкатывается как статический web-export через EAS Hosting.

Собрать production export:

```bash
npx expo export --platform web --output-dir dist-prod
```

Загрузить export в production:

```bash
npx eas-cli@latest deploy --prod --export-dir dist-prod
```

`dist-prod/` - локальный артефакт сборки. Он создается перед deploy, загружается в EAS Hosting и не должен попадать в Git.
